记录自己关于上下文工程的一些理解，以及在工程中的实践。

## 上下文压缩

### 大型工具结果剪裁

在工具每次调用返回结果时进行检查，如果超过指定的上下文大小，触发剪裁。

- 将返回结果序列化到文件中
- 生成文件路径引用和结果前10行的预览
- 用简短引用替换原结果，加入上下文
- 后续访问时，根据引用找到结果文件，按需读取

### 大型工具输入卸载

上下文占用量达到模型最大输入窗口的指定临界点，触发 卸载。清理的目标是历史中 write_file 或 edit_file 这类“写入型”工具调用的参数

- 遍历历史中的write_file 或 edit_file 这类“写入型”工具调用
- 卸载原工具调用的输入参数，参数写入文件中，替换为文件路径引用

### 对话摘要

当前两层卸载策略已无法腾出足够空间，或上下文已接近极限时，会触发最后的保障——对话摘要。

- 默认触发阈值：通常是模型最大输入窗口的 85% 或总 token 数达到 170,000 时。
- 默认保留策略：保留最近 6条消息 或 总 token 数的 10% 作为“新鲜”上下文。

> 双组件协同摘要

摘要流程由两个协同组件共同完成：

上下文摘要生成：系统调用专用LLM，读取即将移出窗口的对话历史，生成结构化摘要，内容涵盖用户核心意图、已完成的任务成果（如代码、文档等）以及后续计划要点。

原始数据持久化：被移出的完整原始对话消息经序列化后，以永久文件形式存储至文件系统（如按会话 ID 命名的日志文件），供日后审计、回溯或恢复使用。

> 上下文重组与资源释放

生成的摘要替换大量历史消息，显著释放模型的上下文窗口空间。同时，依据保留策略（如保留最近 N 条消息）将最新的对话内容与摘要共同置于上下文中，确保后续交互的连贯性和即时性。

> 可选的主动触发机制

除基于上下文长度或轮次等被动条件自动触发外，系统还可向代理（Agent）提供可调用的压缩工具（如 compact_conversation）。代理能够在适当时机（例如完成一项子任务后）主动调用该工具，提前执行摘要，实现主动式上下文管理，增强灵活性。
